¿Qué es INKA?

Motor JavaScript para aventuras gráficas educativas, render-agnóstico

INKA es un motor de aventura gráfica point-and-click escrito en JavaScript puro. Es el cerebro del juego: no dibuja nada directamente. Dirige la lógica, los estados, los diálogos, el inventario y la navegación hablando siempre a través de una interfaz IRenderDriver intercambiable.

Está diseñado específicamente para proyectos educativos: incluye telemetría xAPI compatible con Articulate Storyline/LMS y un plugin de IA pedagógica con feedback adaptativo.

Principios de diseño

  • Render-agnóstico: el motor nunca llama a createjs.* ni PIXI.* directamente. Todo pasa por INKA.Render.
  • Dirigido por datos: el comportamiento completo del juego se describe en un objeto GAME_DATA. No hay código por escena.
  • Canal de errores único: INKA.Log centraliza todos los avisos, con overlay opcional en pantalla.
  • Plugins como ciudadanos de primera clase: telemetría e IA son plugins que el motor inicializa, itera en cada tick y destruye ordenadamente.
  • Animaciones que esperan: las cutscenes no dependen de adivinar milisegundos; la acción animar puede esperar a que el clip termine.

Arquitectura del sistema

Capas, responsabilidades y flujo de datos

📦 Capa 1 — Drivers de render (opcional antes del núcleo)

INKA-driver-animate.js o INKA-driver-spine.js — implementan IRenderDriver. Se cargan antes que el núcleo.

⚙️ Capa 2 — Núcleo (INKA-nucleo.js)

Motor completo: escenas, diálogos, inventario, NPCs, cámara, pathfinding, localización, guardado. Expone la variable global INKA.

🔌 Capa 3 — Plugins

inka-plugin-telemetria.js y inka-plugin-ia.js. Se registran automáticamente al cargarse si INKA ya existe.

🛠️ Capa 4 — Herramientas de autor

inka-dialogos.html — Editor visual de conversaciones. Herramienta standalone, no entra en producción.

Flujo de un clic del jugador

Clic en pantalla
  └─ Input.capturar() — ¿hay pila de bloqueo? (diálogo, cutscene, visor)
       └─ NO → Character.caminarA(x, y)
            └─ Nav.calcularRuta() — pathfinding si hay polígono
                 └─ Render.mover(mc, destino, ms, easing, cb)
                      └─ [Driver Animate] createjs.Tween.to(...)
                         [Driver Spine]   ticker propio sobre PIXI.app.ticker

Ciclo de vida de una escena

Scene.ir(id)
  ├─ Mundo.asegurar(id, cfg)       // crea nodo si es dinámico
  ├─ Shooter/Danger.forzarFin()
  ├─ Dialogue.cerrar()             // drena cola, libera pila Input
  ├─ Depth.soltarPersonaje()       // z-order de escena anterior
  ├─ actualMC.visible = false
  ├─ Nav.preparar(cfg.walkable)    // grafo de esta escena
  ├─ Regiones.preparar(cfg)
  ├─ Actores.limpiar() + crear()   // NPCs de la escena nueva
  ├─ Depth.preparar(cfg)           // walk-behind de esta escena
  ├─ Character.colocar(m.x, m.y)
  ├─ Camara.preparar(cfg)
  ├─ Sound.musica(cfg.musica)
  ├─ Plugins.alEntrarEscena(cfg)   // hooks de plugins
  └─ Game.events.emit('escena:cambiada', {id})

Render Driver (Render)

Punto único de indirección hacia la librería gráfica. Game.init() lo asigna según { driver: "animate" | "spine" }. Todo el resto del motor llama a Render.<método> y nunca a createjs.* ni a lib.*.

// Selección del driver en init:
if (opts.driver === "spine")   Render = new global.INKA_DriverSpine();
else                           Render = new global.INKA_DriverAnimate();
Render.init(root, opts);

Sistema de Log

Todos los errores y avisos pasan por INKA.Log. En modo debug aparecen en consola; con debugOverlay: true también en un overlay sobre el canvas.

MétodoDescripción
INKA.Log.info(...)Solo se emite si debug: true en GAME_DATA
INKA.Log.warn(...)Siempre a consola (problema real pero no fatal)
INKA.Log.error(...)Siempre a consola; queda en historial

Config en GAME_DATA: debug: true, debugOverlay: true. El overlay muestra los últimos 14 mensajes sobre el canvas.

Utilidades (Util)

  • Util.clamp(v, min, max) — recorta valor al rango
  • Util.porId(arr, id) — busca objeto por campo id en array
  • Util.finDeSegmento(mc, label) — devuelve {start, end} en frames del segmento de un MovieClip de Animate que comienza en label
  • Util.puntoEnPoligono(px, py, poly) — ray-casting para walkbox
  • Util.clampPoligono(px, py, poly) — proyecta punto al borde del polígono caminable
  • Util.clonar(v) — deep-copy via JSON (con fallback a referencia si falla)
  • Util.texto(clave, params) — alias de INKA.texto() para localización

Pathfinding (Nav)

Se activa automáticamente cuando la escena tiene walkable.poligono o walkable.obstaculos. Si el suelo es convexo y sin obstáculos, el personaje camina en línea recta (sin overhead).

  • Grafo de visibilidad sobre el polígono caminable y sus obstáculos
  • Algoritmo de búsqueda: Dijkstra (sin heurística, óptimo para espacios acotados)
  • Los puntos de clic fuera del walkable se proyectan al borde más cercano antes de calcular la ruta
  • Config por escena: walkable: { poligono: [{x,y},...], obstaculos: [[{x,y},...]] }

Walk-Behind / Z-Order (Depth)

Permite que el personaje y los NPCs pasen detrás de objetos de decorado (columnas, muebles, puertas).

  • Se activa por escena con profundidad: { objetos: ["columna", "mostrador"] }
  • Ordena personaje y actores por su coordenada Y en cada tick
  • Requiere que la escena esté posicionada en (0,0) — el driver mueve el nodo de escena, no los hijos
  • Sin esta config, el personaje siempre va sobre la escena (comportamiento clásico)

Reloj de juego (Reloj)

Reloj de tiempo de juego (no real) que dispara acciones al cruzar horas configuradas.

// Config en GAME_DATA:
reloj: {
  horaInicial: "08:00",
  velocidad: 60,    // 1 min real = 60 min juego
  agenda: [
    { hora: "20:00", acciones: [{ tipo: "sonido", id: "noche" }] }
  ]
}

NPCs (Actores)

Los NPCs reusan todo el motor de movimiento (pathfinding, dirección, profundidad). Pueden hablar, patrullar por waypoints y obedecer cutscenes.

// Config por escena en GAME_DATA.escenas[i].actores:
actores: [{
  id: "posadero", instancia: "npcPosadero", nombre: "Don Pedro",
  animaciones: { idle: "idle", caminar: "walk" },
  inicio: { x: 400, y: 300 },
  ruta: [{ x: 200, y: 300 }, { x: 400, y: 300 }],
  rutaLoop: true
}]

Reglas reactivas (v2.6)

  • Condiciones: cerca, lejos, siFlag, siNoFlag, siObjeto, ventanaHoraria, fn
  • Efectos: huir, patrullar, caminarA, decir, animar, mirar, o cualquier acción
  • Edge-trigger por defecto; configurables como repetir o unaVez

Cámara (Camara)

Permite escenas más grandes que la pantalla. El HUD permanece fijo.

AcciónEfecto
{ tipo:"camara", seguir: true }Sigue al personaje con smooth
{ tipo:"camara", irA: {x,y} }Desplaza cámara a coordenadas
{ tipo:"camara", zoom: 1.5 }Aplica zoom
{ tipo:"camara", temblar: 300 }Shake durante N ms

Config por escena: camara: { ancho, alto, seguir, suavizado, zoom }. El walkable debe cubrir todo el mundo, no solo la pantalla.

Input unificado (Input)

Gestiona mouse, touch y teclado con una pila de bloqueo. Si hay diálogo, visor, teclado o cutscene encima, los clics del mundo se ignoran.

  • Touch real: createjs.Touch.enable() + bloqueo de scroll/zoom del navegador
  • Arrastre e ignorar toques sobre hotspots que dispara el clic de caminar
  • Teclado: ESC (cerrar/avanzar), Espacio/Enter (avanzar diálogo), 1..9 (elegir opción)
  • Revelar interactuables: tecla configurable en PC / INKA.Input.alternarRevelar() desde botón HUD en móvil
  • Acción secundaria "examinar": clic derecho en PC, pulsación larga en móvil
  • Drag & drop de inventario con input.arrastrar: true; fantasma en hud.cursorObjeto

Animación (Animation)

Gestiona todos los estados visuales del personaje y los actores. Llama siempre a Render.aplicarAnimacion() o Render.reproducirAnimacion(), nunca a gotoAndStop directamente.

  • La acción animar puede bloquear la cadena de acciones hasta que la animación termine (esperar: true)
  • Los estados se definen en GAME_DATA.personaje.estados[] con su label (frame-label en Animate, nombre de animación en Spine)
  • Escala configurable: si el arte del personaje mira a la izquierda, signoEscala: -1

Inventario (Inventory)

  • Objetos con id, imagen, nombre, descripción, acciones al examinar/usar/combinar
  • Slots visuales en HUD configurables por nombre de instancia
  • Combinación de objetos: { tipo:"mezclar", obj1, obj2, resultado }
  • Persistido en guardado (SaveLoad v3)

Localización (Idioma)

Sistema de paquetes por idioma con cambio en runtime. Compatible con el GAME_DATA.i18n anterior.

APIDescripción
INKA.Idioma.set("en")Cambia idioma en tiempo de ejecución
INKA.Idioma.cargarUrl("en", url)Carga paquete desde archivo JSON externo
INKA.Idioma.cargarTodos({es:"...",en:"..."})Carga varios paquetes a la vez
INKA.texto(clave, params)Interpolación {var} y plural simple (one/other)

Diálogos (Dialogue)

Sistema de diálogos completo, nativo en el núcleo. Reemplaza completamente a inka-opciones.js.

  • Máquina de escribir configurable por velocidad (0 = instantáneo)
  • Un clic completa la línea; el siguiente avanza. Auto-avance opcional por línea
  • Líneas con hablante, retrato, voz (vozId) y velocidad/auto-avance individuales
  • Árbol de conversaciones (GAME_DATA.conversaciones): nodos con líneas/acciones/opciones
  • Opciones con condiciones: siFlag, siNoFlag, siObjeto, siNoObjeto, fnCondicion
  • Opciones "una vez", ocultar o agrisar "ya dicho" (persistido en flags → sobrevive al save)
  • Render por panel propio (opcion0..N) o dinámico (texto del motor)
  • Cola segura: dos disparos casi simultáneos no se pisan

Escenas (Scene)

Controla la escena activa: transiciones, hotspots, regiones, marcadores, sonido y HUD.

  • Scene.ir(id, marcador) — transición completa con limpieza del estado anterior
  • Scene.encontrar(nombre) — busca nodo por nombre en la escena activa o en el root
  • Scene.marcador(nombre) — devuelve coordenadas de un marcador de la escena
  • La acción irEscena corta la cadena de acciones en curso antes de transicionar

Guardado (SaveLoad v3)

El guardado más completo hasta la fecha. Bloquea durante diálogo, cutscene, peligro o minijuego.

APIDescripción
SaveLoad.guardar(slot)Captura estado completo en localStorage
SaveLoad.cargar(slot)Restaura estado completo
SaveLoad.listar()Lista slots con metadatos
SaveLoad.hay(slot)Boolean — ¿existe guardado en ese slot?
SaveLoad.borrar(slot)Elimina guardado

Captura: flags, inventario, selección, escena, posición+orientación del personaje, posición/orientación/waypoint-de-patrulla de cada NPC, reloj de juego y cámara (zoom + centro).

Sistema de Plugins (Plugins)

Los plugins se registran con INKA.registerPlugin(def) antes o después de Game.init(). El motor los inicializa, ejecuta sus hooks y los destruye ordenadamente.

INKA.registerPlugin({
  nombre: "miPlugin",
  requiere: ["telemetria"],      // dependencias
  acciones: {
    miAccion: function(a, inka, seguir) {
      // return true si es asíncrono
      seguir();
    }
  },
  api: { metodo: function() {} },  // disponible en INKA.miPlugin.*
  init: function(inka) {},
  alEntrarEscena: function(cfg, inka) {},
  tick: function(dt, inka) {},
  destroy: function(inka) {}
});

Orquestador (Game)

Punto de entrada único. Gestiona el loop de tick, el bus de eventos y el ciclo de vida global.

// Modo Animate:
INKA.Game.init(this, GAME_DATA, { driver: "animate", lib: lib });

// Modo Spine:
await INKA.Game.init(root, GAME_DATA, { driver: "spine", pixiApp: app });

El bus de eventos (INKA.Game.events) emite: escena:cambiada, opcion:elegida, conversacion:inicio/fin, objeto:examinado/recogido, region:entrar, juego:ganado/perdido/evaluacion, aprendizaje:objetivo, y más.

Driver Animate (CreateJS)

INKA-driver-animate__1_.js · 11 KB · Estable

Implementa IRenderDriver usando EaselJS/TweenJS/SoundJS. Comportamiento idéntico bit a bit al motor consolidado v2.6. Los symbols ya están arrastrados en el escenario de Adobe Animate antes de Game.init().

✅
requiereNodosPreexistentes = trueLos nodos deben existir en el escenario de Animate antes de que el motor los busque. La validación temprana avisa si faltan.

Ciclo de vida

  • init(stageRoot, opts): habilita Touch si es soportado, configura createjs.Ticker.framerate (default 30 fps), registra el handler de tick
  • onTick(fn) / offTick(token): suscripción al loop de Animate
  • destroy(): desregistra el ticker, detiene todos los tweens con createjs.Tween.removeAllTweens()

Animación

  • reproducirAnimacion(nodo, label, onComplete): gotoAndPlay(label) + monitoreo frame-a-frame con Util.finDeSegmento() para detectar el fin del segmento
  • mover(nodo, props, ms, easing, cb): createjs.Tween.get(nodo).to(props, ms, ease).call(cb)

Driver Spine (PixiJS v8)

INKA-driver-spine__2_.js · 24 KB · Estable

Implementa IRenderDriver usando PixiJS v8 + @esotericsoftware/spine-pixi-v8 + Howler.js. Los esqueletos se cargan por código desde GAME_DATA.spineAssets; no hay escenario de Animate.

✅
requiereNodosPreexistentes = falseLos nodos se crean programáticamente desde los datos del asset. La validación temprana de "MovieClip ya existente" no aplica.

Dependencias (cargar antes)

pixi.js@8 @esotericsoftware/spine-pixi-v8 howler@2

Ciclo de vida

  • init(stageRoot, opts): requiere opts.pixiApp. Aplica compatScaleProps() al root para exponer scaleX/scaleY/numChildren compatibles con Animate. Expone root.stage.canvas para que Input lo encuentre.
  • onTick / offTick: suscripción via app.ticker.add/remove()
  • destroy(): remueve el token del ticker de Pixi

Carga de assets

// En GAME_DATA:
spineAssets: {
  esqueletos: [
    { alias: "heroeData",  src: "assets/heroe.skel"  },
    { alias: "heroeAtlas", src: "assets/heroe.atlas" }

  imagenes: [
    { alias: "fondoTaberna", src: "assets/fondos/taberna.png" }
  ]

// El driver llama:
await driver.precargar(GAME_DATA.spineAssets);

Sistema de eventos (wrapper CreateJS→Pixi)

El motor usa nombres de eventos CreateJS; el driver los traduce transparentemente a Pixi v8 via un mapa interno y wrappers on()/off() por container.

CreateJSPixi v8Notas
clickpointerupFiltrado por umbral de arrastre (12 px)
rollover/rolloutpointerover/outDirecto
mousedownpointerdownDirecto
pressmovepointermoveMientras se mantiene pulsado
pressuppointerupSolo si el puntero se movió >12px
ℹ️
compatScaleProps() Se aplica a cada container creado para exponer scaleX/scaleY como getters/setters sobre scale.x/scale.y, y numChildren sobre children.length. Hace el código del motor transparente entre drivers.

Animación Spine

  • aplicarAnimacion(nodo, nombre): sk.state.setAnimation(0, nombre, true) (loop)
  • reproducirAnimacion(nodo, nombre, onComplete): setAnimation(0, nombre, false) + listener complete del AnimationState. Más confiable que escanear currentFrame.
  • Si el esqueleto no tiene la animación solicitada, loga un warning y llama onComplete para no bloquear la cadena
  • Los nombres de animación deben coincidir con los del editor de Spine (no con frame-labels)

Tween propio

Sin dependencia extra. Corre sobre el ticker de Pixi con interpolación y easings básicos.

lineareaseIneaseOuteaseInOut

La función mover(nodo, props, ms, easingId, onComplete) devuelve un handle con propiedad cancelado. detenerMovimiento(handle) cancela el tween y lo saca del ticker.

Plugin Telemetría v3.1

inka-plugin-telemetria__2_.js · 25 KB · Estable

Plugin de telemetría educativa con doble modo de operación: xAPI compatible con Articulate/LMS, o standalone con exportación JSON/CSV y endpoint HTTP propio.

Modos de operación (autodetectados)

🎓 Modo xAPI (Articulate)

Se activa si window.ADL y window.ADL.XAPIWrapper están disponibles. Storyline los inyecta automáticamente al correr en un LMS. Envía statements granulares en tiempo real al LRS configurado.

📁 Modo Standalone

Guarda todo en memoria. Exporta JSON/CSV para análisis manual. Compatible con endpoint HTTP propio (autoenviar + intervalo).

Capa xAPI — Statements emitidos

VerboCuándo se emite
experiencedAl entrar a cada escena nueva
answeredAl elegir cada opción de diálogo
progressedAl completar un objetivo pedagógico (aprendizaje:objetivo)
completedAl cerrar el juego (beforeunload / destroy)
passed / failedSegún si el score supera umbralAprobacion

Modo standalone — Configuración

  actividadNombre:  "Aventura de fracciones",

Plugin IA v2.1

inka-plugin-ia__3_.js · 20 KB · Estable

Adaptador pluggable para traducción, análisis pedagógico, generación de diálogo y feedback adaptativo. Lee telemetría si el plugin está activo; nunca escribe en él.

Backend externo (window.INKA_IA)

El plugin no trae ningún modelo de IA. Delega en un objeto window.INKA_IA que el proyecto debe proveer antes de cargar el juego.

window.INKA_IA = {
  // Traduce texto de idioma 'de' a 'a'
  traducir: async (texto, de, a) => "translated text",

  // Analiza decisiones pedagógicas del estudiante
  analizarDecision: async (contexto) => ({
    resumen: "El estudiante...",
    recomendacion: "Se sugiere..."
  }),

  // Genera texto libre con contexto
  generarDialogo: async (prompt, contexto) => "texto generado"
};

Si una capacidad no está configurada, el plugin la declara como no disponible y el motor continúa sin interrupciones.

Contexto enriquecido con telemetría

Cuando el plugin de telemetría está activo, el contexto enviado a la IA incluye datos reales de sesión:

  • Últimas 10 decisiones del estudiante (con correcto/incorrecto y escena)
  • Últimos 20 eventos de sesión
  • Tiempo acumulado por escena
  • Objetivos pedagógicos cumplidos
  • Duración total de la sesión en ms
  • Número de visitas a la escena actual

Si telemetría no está disponible, cae a un historial local de decisiones capturado directamente.

Acciones disponibles desde GAME_DATA

// Generar pista y guardar en State:
{ tipo: "iaPista", prompt: "Dame una pista sobre...",
  guardarEn: "pista", timeout: 8000 }

// Generar feedback de la escena actual:
{ tipo: "iaFeedback", guardarEn: "feedbackEscena",
  timeout: 10000 }

Ambas acciones son asíncronas (devuelven true al motor). Usan una cola interna que serializa las llamadas al backend para no solaparlas.

Timeout wrapper

Toda llamada al backend está envuelta en conTimeout(promesa, ms, fallback). Si el proveedor no responde a tiempo, la promesa resuelve con el fallback (nunca rechaza) y la cadena del motor continúa.

API pública — INKA.ia.*

MétodoDescripción
.traducir(texto, de, a)→ Promise<string>. Fallback = texto original
.generar(prompt, ctx)→ Promise<string>. Contexto enriquecido con telemetría
.analizar()→ Promise<{resumen, recomendacion}>. Timeout mín. 5s
.feedbackEscena(id, timeout)→ Promise<string>. Feedback pedagógico para una escena
.decisiones()Copia inmutable del historial local de decisiones
.tiene(cap)Boolean — ¿el proveedor tiene esta capacidad?

Feedback automático por escena

Si GAME_DATA.ia.feedbackAutomatico: true, el plugin genera feedback al entrar a cada escena (excepto la primera visita, que aún no tiene contexto).

// En GAME_DATA:
ia: {
  feedbackAutomatico:  true,
  feedbackGuardarEn:   "feedbackEscena",  // flag de State
  feedbackTimeout:     10000,
  maxDecisionesAnalisis: 50,
  timeoutDefault:      8000
}

Issues conocidos en v2.1

⚠️
F13 — Race condition: IA lee telemetría antes de que el hook de escena haya disparado Plugins.alEntrarEscena() itera la lista en orden de registro. Si el plugin IA fue registrado antes que telemetría, su hook alEntrarEscena corre antes de que telemetría registre el evento "escena" de la visita actual. El setTimeout(0) en el código mitiga esto cediendo el turno, pero no garantiza el orden si hay operaciones asíncronas intermedias. Estado: parcialmente mitigado.
✅
Correcciones aplicadas en v2.1 Precedencia &&/|| corregida en contextoEscena(). Catch vacío ahora loga en modo debug. camel() eliminada (era identidad sin efecto). Timeouts consistentes en analizar(). Guardia _activo añadida en generarFeedbackEscena().

Issues pendientes

2 issues residuales identificados post-parcheo

⚠️
F-T1 — Null guard en enviarHTTP() del plugin Telemetría Archivo: inka-plugin-telemetria.js
Ubicación: función enviarHTTP(), interacción con el timer automático _timerEnvio
Descripción: Si destroy() se llama y el clearInterval(_timerEnvio) no alcanza a ejecutarse antes de que el timer dispare una vez más, enviarHTTP(inka) puede ser llamado con S = null. La guarda if (!S) al inicio de la función cubre esto, pero el _inka del cierre ya fue puesto a null en destroy(), por lo que el cfg(inka) interno recibiría un valor nulo.
Impacto: Bajo — solo ocurre en la ventana de milisegundos entre clearInterval y la última ejecución del timer.
Fix sugerido: Añadir if (!_inka || !S) return Promise.resolve(false); al inicio de enviarHTTP().
⚠️
F13 — Race condition IA ↔ Telemetría en alEntrarEscena() Archivo: inka-plugin-ia.js
Ubicación: hook alEntrarEscena(), función Plugins.alEntrarEscena() del núcleo
Descripción: Si el plugin IA se registró antes que telemetría, su hook alEntrarEscena se ejecuta primero. El setTimeout(0) cede el turno a handlers síncronos pendientes, pero si el hook de telemetría suscribir → on("escena:cambiada") es también asíncrono o depende de un event-loop posterior, el conteo de visitas puede estar una visita por debajo.
Impacto: Medio — el feedback automático nunca se genera en revisitas cuando el orden de registro es IA → Telemetría.
Fix sugerido: Registrar siempre telemetría antes que IA (ya documentado como requisito), o que el núcleo garantice el orden de llamada de hooks según el campo requiere.
